# 推荐系统学习笔记

## 集中常见的消费指标（短期指标）
### 1. 点击率（CTR）

$$
\text{点击率} = \frac{\text{点击次数}}{\text{曝光次数}}
$$

- 衡量：内容吸引用户点击的能力

---

### 2. 点赞率

$$
\text{点赞率} = \frac{\text{点赞次数}}{\text{点击次数}}
$$

- 衡量：用户对内容的认可程度

---

### 3. 收藏率

$$
\text{收藏率} = \frac{\text{收藏次数}}{\text{点击次数}}
$$

- 衡量：内容的实用价值或长期参考价值

---

### 4. 转发率

$$
\text{转发率} = \frac{\text{转发次数}}{\text{点击次数}}
$$

- 衡量：内容的传播潜力和社交价值

---

### 5. 阅读完成率

$$
\text{阅读完成率} = \frac{\text{滑动到底次数}}{\text{点击次数}} \times f(\text{笔记长度})
$$

- 衡量：内容的持续吸引力和信息密度
- 说明：\( f(笔记长度) \) 为归一化函数，用于平衡长短内容可比性

---

## 消费指标汇总表

| 指标    | 分母   | 分子    | 衡量维度 |
| ----- | ---- | ----- | ---- |
| 点击率   | 曝光次数 | 点击次数  | 吸引力  |
| 点赞率   | 点击次数 | 点赞次数  | 认可度  |
| 收藏率   | 点击次数 | 收藏次数  | 实用价值 |
| 转发率   | 点击次数 | 转发次数  | 传播力  |
| 阅读完成率 | 点击次数 | 滑到底次数 | 内容粘性 |


---

# 北极星指标（最关键指标）

## 1. 用户规模

- **DAU（日活跃用户数）**：每日使用产品的去重用户数（如果我是一个用户，无论我今天使用了一次小红书还是十次，都算你今天贡献了一个DAU）
- **MAU（月活跃用户数）**：每月使用产品的去重用户数（如果我是一个用户，在这个月，无论我使用了一次小红书还是每天都使用小红书，都算你这个月贡献了一个DAU）

> 反映产品的用户基础和覆盖广度

---

## 2. 消费

- **人均使用推荐时长**：用户平均在推荐内容上花费的时间（刷了一个小时的小红书推荐，那我就贡献了一个小时的时长）
- **人均阅读笔记数量**：用户平均浏览/阅读的笔记篇数（看了20篇小红书笔记，我就贡献了20篇的笔记阅读量）
**通常点击率与时长和阅读数量的涨跌是一致的，若出现偏差则以北极星指标为准**

> 反映推荐系统对用户的吸引力和留存能力

---

## 3. 发布

- **发布渗透率**：发布内容的用户数 / 总活跃用户数
- **人均发布量**：总发布内容数 / 活跃用户数

> 反映内容生态的健康度和用户创作意愿

---

## 北极星指标汇总

| 维度 | 核心指标 | 衡量目的 |
|------|----------|----------|
| 用户规模 | DAU、MAU | 产品覆盖与增长 |
| 消费 | 人均推荐时长、人均阅读笔记数 | 用户粘性与推荐效果 |
| 发布 | 发布渗透率、人均发布量 | 内容生态活跃度 |

---
## 实验流程三阶段

### 1. 离线实验

- **方式**：收集历史数据，在历史数据上进行训练和测试
- **特点**：算法**未部署**到实际产品中，**没有**与真实用户发生交互
- **优点**：快速迭代、成本低、可复现
- **缺点**：无法反映线上真实环境中的用户反馈

---

### 2. 小流量 AB 测试

- **方式**：将算法部署到实际产品中，面向**小部分用户**（小流量）进行线上实验
- **特点**：真实用户与算法进行实际交互，收集真实反馈数据
- **目的**：在可控范围内验证算法效果，对比实验组与对照组的表现
- **注意**：需保证统计显著性后再决策

---

### 3. 全流量上线

- **方式**：算法通过 AB 测试验证通过后，逐步或一次性部署到**全部用户**
- **特点**：所有用户均可体验到新算法带来的推荐结果
- **后续**：仍需持续监控核心指标，防止线上环境变化导致效果衰减

---
# 2. 推荐系统的链路（Recommendation Pipeline）

> 推荐系统通常不会直接从海量物品中选出最终推荐结果，而是采用**多阶段筛选（Multi-Stage Ranking）的方式，逐步缩小候选集，提高推荐效率和准确率。

推荐系统的整体流程如下：

```text
几亿物品
    │
    ▼
┌─────────┐
│  召回   │
└─────────┘
    │
    ▼
几千物品
    │
    ▼
┌─────────┐
│  粗排   │
└─────────┘
    │
    ▼
几百物品
    │
    ▼
┌─────────┐
│  精排   │
└─────────┘
    │
    ▼
几百物品
    │
    ▼
┌─────────┐
│  重排   │
└─────────┘
    │
    ▼
几十物品
```

---

# 1.1 为什么推荐系统要分阶段？

假设某个电商平台有：

- 5 亿件商品
- 1000 万用户

如果用户每刷新一次页面，都要计算：

```
1000万 × 5亿
```

几乎是不可能完成的。

因此，推荐系统采用：

> **逐层筛选（Cascade Pipeline）**

每一层都减少候选商品数量。

例如：

```
5亿

↓

5000

↓

500

↓

100

↓

20
```

这样可以极大减少计算量。

---

# 1.2 第一阶段——召回（Recall）

## 什么是召回？

召回（Recall）就是：

> **从海量物品中快速找出用户可能感兴趣的一小部分商品。**

例如：

淘宝有：

```
5亿商品
```

用户：

```
张三
```

召回之后：

```
5000个候选商品
```

后续模型只需要处理：

```
5000
```

而不是：

```
5亿
```

---

## 召回阶段的特点

特点：

- 数据量最大
- 模型速度最快
- 不要求特别准确
- 宁可多召回，也不要漏掉用户可能喜欢的商品

因此：

> **召回关注的是覆盖率（Recall）。**

---

## 常见召回算法

### （1）热门召回（Hot Recall）

直接推荐热门商品。

例如：

```
销量排行榜

热门电影

爆款商品
```

优点：

- 实现简单
- 速度快

缺点：

- 个性化能力弱

---

### （2）协同过滤（Collaborative Filtering）

根据：

```
相似用户

↓

推荐商品
```

或者：

```
相似商品

↓

推荐商品
```

例如：

```
你喜欢《哈利波特》

↓

其他喜欢《哈利波特》的人

↓

还喜欢《魔戒》
```

于是推荐：

```
《魔戒》
```

---

### （3）Embedding 召回

利用深度学习训练：

```
用户向量

商品向量
```

然后计算：

```
向量相似度
```

这是目前工业界最常用的方法。

例如：

- YouTube DNN
- DSSM
- 双塔模型（Two-Tower）

---

# 1.3 第二阶段——粗排（Pre-Ranking）

## 什么是粗排？

召回得到：

```
5000个商品
```

粗排进一步筛选：

```
5000

↓

500
```

粗排开始使用：

> **较复杂的机器学习模型。**

---

## 粗排阶段特点

要求：

- 比召回准确
- 比精排快
- 可以使用简单神经网络

例如：

```
Wide&Deep

DeepFM

DIN
```

---

## 输入

粗排输入通常包括：

```
用户特征

商品特征

统计特征/上下文特征
```

例如：

```
年龄

性别

点击历史

时间

地点
```

---

## 输出

输出：

```
每个商品一个分数
```

然后：

```
排序

↓

Top500
```
![[Pasted image 20260711213652.png]]

---

# 1.4 第三阶段——精排（Ranking）

## 什么是精排？

精排是：

> **推荐系统最核心的阶段。**

输入：

```
500个商品
```

输出：

```
最终排序结果
```

---

## 精排阶段特点

可以使用：

- 深度学习
- Transformer
- Attention
- 多任务学习

例如：

```
DIN

DIEN

AutoInt

DCN

xDeepFM

SASRec

BERT4Rec
```

---

# 1.5 第四阶段——重排（Re-Ranking）

## 什么是重排？

精排结束后：

```
Top100
```

最后：

```
Top20
```

需要进一步优化：

不仅仅考虑：

```
点击率
```

还需要考虑：

- 多样性（Diversity）
- 新颖性（Novelty）
- 公平性（Fairness）
- 用户体验（User Experience）

---

## 为什么需要重排？

例如：

用户喜欢：

```
篮球
```

精排结果：

```
篮球

篮球

篮球

篮球

篮球
```

用户会觉得：

```
推荐太单一
```

于是：

重排之后：

```
篮球

足球

NBA

运动鞋

健身
```

推荐更加丰富。

---

## 重排常用方法

例如：

- MMR（Maximum Marginal Relevance）
- DPP（Determinantal Point Process）
- 多样性优化
- 规则排序
- 广告插入

---

# 1.6 推荐链路总结

| 阶段 | 输入 | 输出 | 主要目标 | 常见模型 |
|------|------|------|----------|----------|
| 召回 | 几亿商品 | 几千商品 | 快速找到候选集 | 协同过滤、Embedding、双塔模型 |
| 粗排 | 几千商品 | 几百商品 | 快速评分 | Wide&Deep、DeepFM、DIN |
| 精排 | 几百商品 | 几百商品（排序） | 精确预测用户兴趣 | DCN、DIEN、SASRec、BERT4Rec |
| 重排 | 几百商品 | 几十商品 | 优化用户体验 | MMR、DPP、多样性优化 |

---

# 1.7 推荐系统整体流程

```text
海量商品（亿级）
        │
        ▼
    召回（Recall）
        │
        ▼
候选商品（千级）
        │
        ▼
   粗排（Pre-Ranking）

        ▼

        │

```

# 2. 基于物品的协同过滤（Item-Based Collaborative Filtering，ItemCF）

> ItemCF（Item-Based Collaborative Filtering，基于物品的协同过滤）是一种经典的推荐算法，其核心思想是：
>
> **如果两个物品经常被相同的用户喜欢，那么这两个物品就是相似的。**

---

# 2.1 ItemCF 的基本思想

ItemCF 不关注"用户之间是否相似"，而关注：

> **物品之间是否相似。**

例如：

```
我喜欢看《笑傲江湖》

↓

《笑傲江湖》和《鹿鼎记》很相似

↓

我没有看过《鹿鼎记》

↓

推荐《鹿鼎记》
```

整个推荐过程可以概括为：

```
用户喜欢物品A
        │
        ▼
计算与物品A相似的物品
        │
        ▼
过滤已经看过的物品
        │
        ▼
推荐相似物品
```

因此，ItemCF 的推荐依据不是"相似用户"，而是"相似物品"。

---

# 2.2 推荐系统如何知道两个物品相似？

推荐系统并不知道：

```
《笑傲江湖》
```

和

```
《鹿鼎记》
```

都是武侠小说。

它只是根据大量用户的行为统计得到：

例如：

很多用户都有这样的浏览记录：

```
《笑傲江湖》

↓

《鹿鼎记》
```

或者：

```
喜欢《笑傲江湖》的用户

↓

也喜欢《鹿鼎记》
```

于是系统认为：

```
《笑傲江湖》

≈

《鹿鼎记》
```

因此：

> **两个物品拥有越多共同用户，它们越相似。**

---

# 2.3 ItemCF 的基本流程

整个算法主要分为两步。
- 计算候选物品与历史喜欢物品的相似度
- 根据用户已喜欢的物品及相似度，为候选物品计算兴趣分数并排序推荐。


![[Pasted image 20260711221701.png]]
上图中用户对候选物品的兴趣：2×0.1+1×0.4+4×0.2+3×0.6=3.2
![[Pasted image 20260711221507.png]]
![[Pasted image 20260711221557.png|697]]
|v|就表示同时喜欢着两个物品的用户的个数
![[Pasted image 20260711221624.png]]
其中：

- **user**：用户
- **item**：候选物品
- **itemj**：用户历史喜欢的物品
- **like(user,itemj)**：用户对物品 j 的兴趣（评分、点击次数、浏览次数等）
- **sim(itemj,item)**：两个物品之间的相似度

即：

> **用户兴趣 = 用户历史兴趣 × 物品相似度 的加权求和。**

---

## 示例

假设：

用户历史：

| 电影 | 兴趣值 |
|------|-------|
| A |2|
| B |1|
| C |4|
| D |3|

候选电影：

```
Movie X
```

相似度：

| 已看电影 | 与X相似度 |
|----------|----------|
|A|0.1|
|B|0.4|
|C|0.2|
|D|0.6|

则：

```
score(X)

=

2×0.1

+

1×0.4

+

4×0.2

+

3×0.6

=

0.2+0.4+0.8+1.8

=

3.2
```

最终：

按照 Score 排序。

# 2.4 ItemCF 的优缺点

## 优点

- 算法简单，实现容易。
- 推荐结果稳定，不容易受到单个用户行为影响。
- 推荐解释性强（"因为你喜欢 A，所以推荐与 A 相似的 B"）。
- 商品相似度可以离线计算，提高在线推荐效率。
- 在电影、音乐、图书、电商等领域应用广泛。

---

## 缺点

### ① 冷启动问题

新商品没有用户行为。

无法计算：

```
相似度
```

因此：

不能推荐。

---

### ② 长尾问题

冷门商品：

```
用户太少
```

导致：

```
相似度不准确
```

---

### ③ 兴趣难以扩展

用户喜欢：

```
篮球
```

一直推荐：

```
篮球

篮球

篮球
```

缺乏：

```
多样性
```

---

### ④ 数据稀疏问题

如果：

```
用户

×

商品
```

矩阵非常稀疏。

共同用户很少。

导致：

```
相似度不稳定
```

---

# 2.5 ItemCF 与 UserCF 的区别

| 比较项 | ItemCF | UserCF |
|---------|--------|--------|
| 核心思想 | 找相似物品 | 找相似用户 |
| 比较对象 | 商品之间 | 用户之间 |
| 推荐依据 | 用户喜欢的相似商品 | 相似用户喜欢的商品 |
| 稳定性 | 高 | 较低 |
| 工业应用 | 更广泛 | 较少 |

---

# 2.6 ItemCF 在工业界的应用

ItemCF 是工业推荐系统的重要组成部分，常用于**召回阶段（Recall）**。

典型应用包括：

- Amazon："购买了该商品的用户还购买了……"
- 淘宝："猜你喜欢"
- 京东："看了又看"
- Netflix："因为你看过……"
- Bilibili："相关推荐"

在工业系统中，通常会先利用 ItemCF 快速召回候选物品，再结合深度学习排序模型（如 DeepFM、DIN、SASRec、BERT4Rec）完成粗排和精排。

# 3. Swing 模型（Swing Recommendation Algorithm）

Swing 是阿里巴巴提出的一种**基于物品的协同过滤改进算法（Improved Item-based Collaborative Filtering）**。

它是在 ItemCF 的基础上发展而来，主要解决 ItemCF 在电商推荐中容易受到**热门物品**、**小圈子用户**影响的问题。

相比 ItemCF，Swing 并没有改变推荐流程，而是**改进了物品相似度的计算方式**。

---

## 3.1 为什么提出 Swing？

### ItemCF 存在的问题

ItemCF 认为：

> 两个物品共同喜欢的人越多，它们越相似。

例如：

```
电影A：《笑傲江湖》

电影B：《鹿鼎记》
```

如果很多用户同时看过：

```
笑傲江湖
鹿鼎记
```

那么：

```
sim(A,B) ↑
```

但是实际中存在一种情况：

```
用户A
用户B
用户C
```

他们三个好友兴趣完全一样：

```
电影A
电影B
电影C
电影D
```

于是：

```
A-B
A-C
A-D
……
```

全部都会得到非常高的相似度。

但是：

这只是一个很小的兴趣圈子。

如果整个网站只有他们三个人喜欢，

并不能说明：

```
A 和 B 真正很相似。
```

因此：

**ItemCF 会高估小圈子带来的相似度。**

---

## 3.2 Swing 的核心思想

Swing 的思想：

> **共同喜欢两个物品的用户，如果彼此太相似，那么他们提供的信息价值就降低。**

也就是说：

不是所有共同用户都应该拥有一样大的权重。

例如：

```
用户1
用户2
```

如果：

他们浏览记录几乎一样：

```
100 个物品
重合 95 个
```

那么：

他们很可能：

```
同一个家庭

或者

同一个兴趣群体
```

他们提供的是

```
重复信息
```

应该降低贡献。

反过来：

如果：

```
两个共同用户

兴趣差异很大

却都喜欢

A
B
```

说明：

```
A 和 B
真的很相关。
```

因此：

Swing 会给予更大的权重。

---

## 3.3 用户重合度（Overlap）

设：

用户u₁，喜欢的物品集合：J₁；用户u₂喜欢的物品集合：J₂，那么两个用户的重合度定义为：
overlap(u₁, u₂)=|J₁ ∩ J₂|
即：**两个用户共同喜欢多少物品。**
用户u₁和u₂的重合度越高，他们就很可能来自一个小圈子，要降低他们的权重。
例如：

```
用户A：

电影1
电影2
电影3
电影4

用户B：

电影2
电影3
电影4
电影5
```

共同喜欢：

```
电影2
电影3
电影4
```

因此：

```
overlap=3
```

说明：

他们兴趣十分接近。

---

## 3.4 Swing 相似度计算

设喜欢i₁的用户集合W₁，喜欢i₂的用户集合W₂，共同用户：V=W₁∩ W₂
Swing 中的两个物品的相似度定义为：
$$
sim(i_1,i_2)
=
\sum_{u_1\in V}
\sum_{u_2\in V}
\frac{1}
{\alpha+overlap(u_1,u_2)}
$$

其中：

- V：同时喜欢两个物品的用户集合
- overlap(u₁, u₂)：两个用户兴趣重合度
- α：平滑参数（一般取 1）


---

## 3.5 Swing 与 ItemCF 的区别

 Swing 与 ItemCF 唯一的区别在于**物品相似度的计算方式**。

- **ItemCF：**
  - 两个物品重合的用户比例高，则判定两个物品相似。

- **Swing：**
  - 额外考虑重合的用户是否来自一个小圈子。
  - 同时喜欢两个物品的用户记作集合：

  $$
  V
  $$

  - 对于集合 $V$ 中的用户 $u_1$ 和 $u_2$，定义两位用户的重合度：
$$
  overlap(u_1,u_2)
  $$

  - 若两个用户的重合度越大，则说明他们越可能来自同一个兴趣小圈子，因此在计算物品相似度时，应降低他们的贡献权重。

# 4. 基于用户的协同过滤（UserCF）

## 4.1 基本思想

- 如果用户 $user_1$ 跟用户 $user_2$ 相似，并且 $user_2$ 喜欢某物品，
- 那么用户 $user_1$ 也很可能喜欢该物品。

## 4.2 预估用户对候选物品的兴趣

预估用户 $user$ 对候选物品 $item$ 的兴趣：

$$
\sum_j sim(user,user_j)\times like(user_j,item)
$$

## 4.3 计算两个用户的相似度

- 把每个用户表示为一个稀疏向量，向量每个元素对应一个物品。
- 相似度 $sim$ 就是两个向量夹角的余弦。
- 用户 $u_1$ 喜欢的物品记作集合 $J_1$。
- 用户 $u_2$ 喜欢的物品记作集合 $J_2$。
- 定义交集

$$
I=J_1\cap J_2
$$

- 两个用户的相似度：
$$
sim(u_1,u_2)=\frac{|I|}{\sqrt{|J_1|\cdot|J_2|}}
$$

## 4.4 降低热门物品权重

- 用户 $u_1$ 喜欢的物品记作集合 $J_1$。
- 用户 $u_2$ 喜欢的物品记作集合 $J_2$。
- 定义交集

$$
I=J_1\cap J_2
$$

- 两个用户的相似度：

$$
sim(u_1,u_2)=
\frac{\displaystyle\sum_{l\in I}\frac{1}{\log(1+n_l)}}
{\sqrt{|J_1|\cdot|J_2|}}
$$

其中 $n_l$表示喜欢物品 $l$ 的用户数量，反映物品的热门程度。

# 4.5 UserCF的基本流程
## 1. 事先做离线计算

### 建立“用户→物品”的索引

- 记录每个用户最近点击、交互过的物品ID。
- 给定任意用户ID，可以找到他近期感兴趣的物品列表。

### 建立“用户→用户”的索引

- 对于每个用户，索引他最相似的k个用户。
- 给定任意用户ID，可以快速找到他最相似的k个用户。

## 2. 线上做召回

1. 给定用户 ID，通过“用户→用户”索引，找到 top-k 相似用户。

2. 对于每个 top-k 相似用户，通过“用户→物品”索引，找到用户近期感兴趣的物品列表（last-n）。

3. 对于召回的 \( nk \) 个相似物品，用公式预估用户对每个物品的兴趣分数。

4. 返回分数最高的 100 个物品，作为召回结果。

# 5. 离散特征处理（Discrete Feature Processing）

## 5.1 基本思想

离散特征处理分为两个步骤。

### （1）建立字典（Dictionary）

把类别映射成序号。

例如：

- 中国 → 1
- 美国 → 2
- 印度 → 3

---

### （2）向量化（Vectorization）

把序号映射成向量。

主要有两种方式：

- **One-hot 编码**：把序号映射成高维稀疏向量。
- **Embedding**：把序号映射成低维稠密向量。

---

## 5.2 One-hot 编码示例

### 例：国籍特征

假设共有 **200** 个国家（类别）。

建立字典：

- 中国 → 1
- 美国 → 2
- 印度 → 3
- ...

采用 **One-hot 编码** 后，用 **200 维稀疏向量** 表示国籍。

例如：

未知：

```text
0 → [0,0,0,0,...,0]
```

中国：

```text
1 → [1,0,0,0,...,0]
```

美国：

```text
2 → [0,1,0,0,...,0]
```

序号对应位置为1，其他全为0。

---

## 5.3 Embedding 示例

### 参数数量

Embedding 的参数数量为：

$$
\text{参数数量}=\text{向量维度}\times\text{类别数量}
$$

例如：

- embedding 输出向量维度为 4；
- 一共有 200 个国家。
- 然后构成一个4×200的参数矩阵，列数就等于（国家）用户数量

则参数数量为：

$$
4 \times 200 = 800
$$

---

### 编程实现

TensorFlow、PyTorch 都提供了 Embedding 层。

Embedding 的参数以矩阵形式保存：

$$
\text{参数矩阵大小}=\text{向量维度}\times\text{类别数量}
$$

输入：

> 输入的是类别对应的序号。

例如：

```text
美国 → 2
```

输出：

> 输出对应的 Embedding 向量。

例如：

```text
美国对应参数矩阵的第 2 列。
```

---

## 5.4 Embedding 的计算过程

Embedding 的计算过程为：

$$
\text{Embedding}=\text{参数矩阵}\times\text{One-Hot向量}
$$

即：

- One-hot 向量只有一个位置为 1；
- 与参数矩阵相乘后，相当于直接取出参数矩阵对应的一列作为该类别的 Embedding 向量。

---

## 5.5 总结

- 离散特征处理主要有两种方式：
  - One-hot 编码
  - Embedding

- 当类别数量很大时，通常使用 **Embedding**。

常见的 Embedding 包括：

- Word Embedding
- 用户 ID Embedding
- 物品 ID Embedding

# 6. 矩阵补充（Matrix Completion）

## 基本思想

矩阵补充（Matrix Completion）是一种经典的推荐算法，其核心思想是：

- 把**用户 ID**、**物品 ID**做 **Embedding**，映射成低维向量。
- 使用两个向量的内积作为用户对物品兴趣的预测值。
- 通过训练，使预测分数尽可能接近真实的用户反馈，从而学习 Embedding 参数。
- 矩阵补充模型存在较多缺点，目前实际推荐系统中使用较少。

---

## 用户与物品 Embedding

- 用户 \(u\) 对应一个向量：

$$
a_u
$$

- 物品 \(i\) 对应一个向量：

$$
b_i
$$

Embedding 层负责学习所有用户和物品对应的向量表示。

---

## 兴趣预测

两个向量的内积作为用户对物品兴趣的预测值：

$$
\langle a_u,b_i\rangle
$$

其中：

- \(a_u\)：用户 \(u\) 的 Embedding 向量；
- \(b_i\)：物品 \(i\) 的 Embedding 向量；
- \(\langle a_u,b_i\rangle\)：两者的内积，即预测兴趣分数。

---

## 模型训练

训练目标：

让预测分数

$$
\langle a_u,b_i\rangle
$$

尽可能拟合真实观测到的用户兴趣分数，从而学习模型的 **Embedding 层参数**。

即不断优化：

- 用户向量 \(a_u\)
- 物品向量 \(b_i\)

使预测结果与真实评分误差最小。

---

## 缺点

矩阵补充（Matrix Completion）存在很多缺点，例如：

- 数据稀疏时效果较差；
- 冷启动问题严重；
- 难以利用用户、物品的丰富特征信息；
- 推荐效果相较于现代深度学习推荐模型较弱。

因此，目前更多作为推荐系统的发展基础，而较少直接应用于工业界。

---
# 7. 双塔模型（Two-Tower Model）

## 基本思想

双塔模型（Two-Tower Model）的核心思想：

- 用户塔（User Tower）输出一个用户向量（User Embedding）。
- 物品塔（Item Tower）输出一个物品向量（Item Embedding）。
- 使用两个向量的**余弦相似度（Cosine Similarity）作为用户对物品兴趣的预测值。

---

## 模型结构

![[Pasted image 20260714163959.png]]
##  模型训练：
### Pointwise（点式学习）

每次训练使用：

- 一个用户（User）
- 一个物品（Item）

物品可以是：

- 正样本（Positive Sample）
- 负样本（Negative Sample）

即：

> 每次用一个用户、一个物品（可正可负）。

---

###  Pairwise（成对学习）

每次训练使用：

- 一个用户（User）
- 一个正样本（Positive Item）
- 一个负样本（Negative Item）

即：

> 每次用一个用户、一个正样本、一个负样本。

Pairwise 的目标是：

使正样本的预测分数高于负样本。

---

### Listwise（列表学习）

每次训练使用：

- 一个用户（User）
- 一个正样本（Positive Item）
- 多个负样本（Negative Items）

即：

> 每次用一个用户、一个正样本、多个负样本。

---
# 8. 正负样本

## 正样本
- 定义：曝光且有点击的用户-物品二元组，表示用户对该物品感兴趣。
- 问题：少部分热门物品占据大部分点击，导致正样本集中于热门物品，产生偏差。
- 解决方案：
  - 过采样（Up-sampling）：让冷门物品样本多次出现。
  - 降采样（Down-sampling）：丢弃部分热门物品样本。

## 负样本分类概述
| 类型 | 来源 | 特点 |
| :---: | :--- | :--- |
| 简单负样本 | 全体物品 / Batch内 | 用户明显不感兴趣，易区分 |
| 困难负样本 | 召回/排序阶段淘汰的物品 | 与正样本相似，难区分 |

## 简单负样本：全体物品抽样
- 依据：未被召回的物品 ≈ 全体物品，大概率是用户不感兴趣的。
- 抽样方式：
  - 均匀抽样：对冷门物品不公平（负样本多为冷门，正样本多为热门）。
  - 非均匀抽样（推荐）：抽样概率与物品热门程度（点击次数）正相关，常用公式：  
    **抽样概率 ∝ (点击次数)^0.75**  
    目的是打压热门物品，缓解偏差。

## 简单负样本：Batch内负样本
- 构造方法：
  - 一个Batch内有 `n` 个正样本（用户-物品对）。
  - 对每个用户，将Batch内其他 `n-1` 个物品作为该用户的负样本。
  - 共产生 `n × (n-1)` 个负样本。
- 特点：这些负样本都属于简单负样本，因为用户对其他用户的点击物品通常不感兴趣。

## 困难负样本
- 来源：
  - 被粗排淘汰的物品（比较困难）。
  - 精排分数靠后的物品（非常困难）。
- 作用：提高模型区分相似物品的能力。
- 分类准确率对比（从易到难）：
  - 全体物品（简单）→ 准确率最高
  - 被粗排淘汰的物品（比较困难）→ 容易分错
  - 精排分数靠后的物品（非常困难）→ 最容易分错

## 常见误区
- 错误做法：将曝光但未点击的物品直接作为召回阶段的负样本。
- 原因：曝光未点击只能说明用户“不感兴趣”，不代表用户“不知道”或“一定不喜欢”，直接作为负样本会引入噪声，尤其在召回阶段不合适。
- 将曝光但未点击的物品可以作为排序模型的负样本。

# 9. 双塔模型的线上召回

## 9.1 背景

在推荐系统的多阶段筛选链路中，**召回** 是第一阶段，目标是从海量物品（如几亿）中快速筛选出用户可能感兴趣的候选集（如几千），为后续的粗排、精排等阶段做准备。

双塔模型是召回阶段最常用的模型之一，因其结构简单、可分离计算、适合大规模检索而广泛应用。

---

## 9.2 双塔模型结构

双塔模型由两个独立的神经网络组成：

- **用户塔（User Tower）**：输入用户特征（如用户 ID、历史行为、画像等），输出用户向量 $u$
- **物品塔（Item Tower）**：输入物品特征（如物品 ID、类别、标签等），输出物品向量 $v$

两个塔的输出向量通过内积计算相似度：

$$
\text{score}(u, v) = u \cdot v
$$

---

## 9.3 线上召回流程

双塔模型在线上召回时，利用其**可分离计算**的特性，分为两个阶段：

### 9.3.1 离线构建物品索引

- 将全量物品通过 **物品塔** 计算得到物品向量 $v$
- 将物品向量存入向量数据库或近似最近邻（ANN）索引（如 Faiss、HNSW）
- 这一步可以离线定期更新（如每天或每小时）

### 9.3.2 线上实时检索

- 当用户请求到来时，将用户特征输入 **用户塔**，实时计算用户向量 $u$
- 使用 $u$ 在已构建的物品索引中进行 **近似最近邻搜索（ANN）**，快速召回 Top-K 个最相似的物品
- 将召回的物品列表传递给下一阶段（粗排）

---

## 9.4 双塔召回的优势

| 优势 | 说明 |
|------|------|
| **计算效率高** | 用户塔在线计算一次，物品塔离线预计算，避免全量计算 |
| **可扩展性强** | 支持亿级物品的快速检索 |
| **结构简单** | 易于训练和部署 |
| **支持多路召回** | 可与其他召回方式（如协同过滤、向量召回）组合使用 |

---

## 9.5 双塔召回的挑战

- **特征交叉不足**：用户塔和物品塔在最后才通过内积交互，缺乏早期特征交叉，可能损失部分表达能力
- **实时性受限**：物品向量需要离线更新，无法实时反映物品的最新变化（如新物品冷启动）
- **近似检索误差**：ANN 检索会引入一定的精度损失

---

## 9.6 与推荐系统链路的关系

在推荐系统链路中，双塔模型通常用于 **召回阶段**：

几亿物品
    │
    ▼
┌─────────┐
│  召回   │  ← 双塔模型在此阶段
└─────────┘
    │
    ▼
几千物品 → 粗排 → 精排 → 重排 → 推荐结果
